Preface

BabyFish is a smart data structure framework.

[Note:This document only discusses the most important functionalities of BabyFish, please see the demo of this framework to know all the functionalities]
Java Part

The Java Part supports smart data structure, it also is the base of JPA part.

Core Functionality: ObjectModel4Java

ObjectModel4Java is the toppest functionality of Java part, it' used to manage all the associations between data model objects, such as one-to-one, one-to-many and many-to-many. Let's see the charm of ObjectModel4Java by the first demo with the bidirectional associations that is combined by List and IndexedReference.

This example shows the classic association between department and employees. One department can retain several employees, each department object uses a collection with element index whose type is java.util.List to retain its employees; conversely, each employee can retain its department object and an index to describe its position in its parent object.

  • The declaration of class "Department" 
    /*
 * The annotation "@org.babyfish.model.Model" means this class uses "ObjectModel4Java"
 * that requires the compilation-time bytecode instrument of BabyFish maven plugin.
 */
@Model
public class Department {

    // The annotation "@org.babyfish.model.Association" declares the association property "employees"
    // It's the mirroring of "Employee.department".
    @Association(opposite = "department")
    private List<Employee;> employees;

    // Get the employee list of current department, 
    // it's unnecessary to support setter because the collection itself can be changed
    // (Of course, it isn't wrong if the setter is supported).
    public List<Employee> getEmployees() {
        return this.employees; 
    }
}
  • The declaration of class "Employee" 
    /*
 * The annotation "@org.babyfish.model.Model" means this class uses "ObjectModel4Java"
 * that requires the compilation-time bytecode instrument of BabyFish maven plugin.
 */
@Model
public class Employee {
    
    // The annotation "@org.babyfish.model.Association" declares the association property "department"
    // It's the mirroring of "Department.employees".
    @Association(opposite = "employees")
    private Department department;

    // The annotation "@org.babyfish.model.IndexOf" means the property "index" is the position of the
    // current object in the "employees" list of parent object which is specified by property "department"
    // (its value should be -1 when the property "department" is null).
    @IndexOf("department")
    private int index;

    public int getIndex() {
        return this.index; 
    }

    public void setIndex(int index) {
        this.index = index; 
    }

    public Department getDepartment() {
        return this.department;
    }

    public void setDepartment(Department department) {
        this.department = department;
    }
}

Any one of association property "Department.employees" and "Employee.department" is mirroring of each other, that means the association between Department and Employee is bidirectional. BabyFish can guarantee the consistency of this bidirectional association, any side can be changed automatically and implicitly when the other side is changed by the programmer.

The annotation "@org.babyfish.model.Model" means these two classes are not simple data classes, they require the maven plugin of BabyFish to do the bytecode instrument work at compilation time, build more bytecode to get stronger ability.

This is smart data structure with chain reaction effect, this is the goal of ObjectModel4Java. In classic projects, developers often declare the data model class like C-style structure(Simple class that contains some private fields and some getters and setters without logic) and shirk the reponsiblity of the complex data association management to business logic developpers. BabyFish want to change this bad present situation and let everyone can develop the smart data structure easily.

Unstable Collection Elements
Why do we need unstable elements

Set and Map are supported by collection framework of almost all languages(Map is called as Dictionary in some languages, such as C#, Objective-C), they are hash structure and red black tree. Hash structure and red black tree can support the ultrafast query performance and good modification performance, but they have a deadly defect: The data objects that has been added into hash structure or red black tree can't be changed. Java language is no exception, Set/Map requires the stable elements(keys) in Java Collection Framework.

This restriction often brings trouble in the data modeling of actual projects. Let's see an example about the many-to-many association between Company and Investor.

  • The declaration of class "Company" 
    /*
 * The annotation "@org.babyfish.model.Model" means this class uses "ObjectModel4Java"
 * that requires the compilation-time bytecode instrument of BabyFish maven plugin.
 */
@Model
public class Company {

    // The annotation "@org.babyfish.model.Scalar" means this property is
    // simple property, not association property.
    @Scalar
    private String name;
        
    /*
     * A company can be jointly funded by multiple investors.
     *
     * The annotation "@org.babyfish.model.Association" means
     * (1) This association property the mirroring of "Investor.companies".
     * (2) The type of this type is ordinary Set, so it works like HashSet
     *
     * The annotation "@org.babyfish.model.ComparatorRule" means this property uses "Investor.name" to
     * (1) Calculate the hash code of each Investor object
     * (2) Check whether two Investor objects are equal
     */
    @Association(opposite = "companies")
    @ComparatorRule(properties = @ComparatorProperty(name = "name"))
    private Set<Investor> investors;

    public String getName() {
        return this.name; 
    }

    public void setName(String name) {
        this.name = name; 
    }

    // Get the investor collection of current company, 
    // it's unnecessary to support setter because the collection itself can be changed
    // (Of course, it isn't wrong if the setter is supported).
    public Set<Investor> getInvestors() {
        return this.investors; 
    }
}
  • The declaration of class "Investor" 
    /*
 * The annotation "@org.babyfish.model.Model" means this class uses "ObjectModel4Java"
 * that requires the compilation-time bytecode instrument of BabyFish maven plugin.
 */
@Model
public class Investor {

    // The annotation "@org.babyfish.model.Scalar" means this property is
    // simple property, not association property.
    @Scalar
    private String name;
        
    /*
     * An investor can invest in multiple companies.
     *
     * The annotation "@org.babyfish.model.Association" means
     * (1) This association property the mirroring of "Company.investors".
     * (2) The type of this type is NavigableSet, so it works like TreeSet
     *
     * The annotation "@org.babyfish.model.ComparatorRule" means this property uses "Company.name" to
     * compare two company objects
     */
    @Association(opposite = "companies")
    @ComparatorRule(properties = @ComparatorProperty(name = "name"))
    private NavigableSet<Company> companies;

    public String getName() {
        return this.name; 
    }

    public void setName(String name) {
        this.name = name; 
    }

    // Get the company collection of current investor, 
    // it's unnecessary to support setter because the collection itself can be changed
    // (Of course, it isn't wrong if the setter is supported).
    public NavigableSet<Company> getCompanies() {
        return this.companies; 
    }
}

These two class declare the bidirectional many-to-many association between companies and investors, this bidirectional association has 2 problems:

  1. The behavior of property "Company.investors" is similar to java.util.HashSet, it uses the property "Investor.name" to calculate the hashCode of Investor objects and check whether two Investor objects are equal. Unfortunately, class "Investor" supports method "setName" so that the property "Investor.name" is unstable, finally, the hashCode of Investor object is unstable. This is clearly not satisfied with the promise about the hash structure requires stable elements.
  2. The behavior of property "Investor.companies" is similar to java.util.TreeSet, it uses the property "Company.name" to compare two Company objects. Unfortunately, class "Company" supports method "setName" so that the property "Company.name" is unstable, finally, the comparsion result of two Company objects is unstable. This is clearly not satisfied with the promise about the red black tree requires stable elements.

In order to resolve these annoying problems, BabyFish extends the interfaces of Java Collection Framework and gives a brand new implementation of collection framework to let Set/Map support unstable elements/keys. That means the data object can still be modified after it has been added into some Sets/Maps. All of those sets/maps will be adjusted automatically when the data object is changed by the user. So the bidirectional association demonstrated by this example can work normally, don't worry.

Details behind the scenes

ObjectModel4Java hides the mechanism behind the scenes by the bytecode instrument at compilation time, let's temporarily ignore ObjectModel4Java and discuss the mechanism of "Unstable collecton elements" behind the scenes by another functionality of this framework: X Collection Framework.

It's necessary to briefly discuss the interface "org.babyfish.collection.EqualityComparator<T>" before discuss unstable collection elements. As we know, TreeSet and TreeMap support both the "java.lang.Comparable" that allows the data class itself to specify how to compare two data objects and "java.util.Comparator" that allows a 3rd-party class to specify how to compare two data objects, but HashSet and HashMap only allows the programmer to override the methods "hashCode" and "equals" of the data class itself to specify how to calculate the hashCode of data object and check whether two data objects are equal. In order to resolve this problem, BabyFish supports a new interface "org.babyfish.collection.EqualityComparator" to allow a 3rd-party class to specify how to calculate the hashCode and check whether two data objects are equals, like the "System.Collections.Generic.IEqualityComparer<T>" of the collection framework of .NET Framework. Its source code is very simple so that it's unnecessary to in-depth discuss it here.

  1. In order to let red black tree support unstable collection elements, BabyFish supports the interface "org.babyfish.collection.FrozenComparator<T>" by exending "java.util.Comparator<T>" 
    package org.babyfish.collection;

import java.util.Comparator<T>;

public interface FrozenComparator<T> extends Comparator<T> {

    // Will be executed by X Collection Framework automatically
    // when "obj" is added into a red black tree
    void freeze(T obj, FrozenContext<T> ctx);

    // Will be executed by X Collection Framework automatically
    // when "obj" is removed from a red black tree
    void unfreeze(T obj, FrozenContext<T> ctx);
}
  2. In a similar way, In order to let hash structure support unstable collection elements, BabyFish supports the interface "org.babyfish.collection.FrozenEqualityComparator<T>" by exending "org.babyfish.collection.EqualityComparator<T>". 
    package org.babyfish.collection;

public interface FrozenEqualityComparator<T> extends EqualityComparator<T> {

    // Will be executed by X Collection Framework automatically
    // when "obj" is added into a hash structure.
    void freeze(T obj, FrozenContext<T> ctx);

    // Will be executed by X Collection Framework automatically
    // when "obj" is added into a hash structure.
    void unfreeze(T obj, FrozenContext<T> ctx);
}

Now, the user can declare a data class to be the collection element type. 

public class FullName{

    private String firstName;

    // Retains all the red black trees and hash structures whose comparators
    // depend on the field "fieldName" of current object by java weak reference.
    // Its default value is null.
    // This field will not be changed by this class, but be changed by other classes,
    // so it uses the default visiblity, not private.
    FrozenContext<ContactInfo> firstNameFrozenContext;

    public String getFirstName() { return this.firstName; }

    public void setFirstName(String firstName) {
        if (!java.util.Objects.equals(this.firstName, firstName)) {
            FrozenContext<ContactInfo> ctx = this.firstNameFrozenContext;

            // Let's all the red-black trees and hash structures that depend on "firstName"
            // of this object suspend freezing this object before the modification, 
            // it will cause this object be removed from those red-black trees or hash structure temporarily.
            FrozenContext.suspendFreezing(ctx, this);

            // Change the data safely becaue no red black tree or hash structure
            // contains this object now.
            this.firstName = firstName;  

            // Let's those red-black trees and hash structure resume freezing this object,
            // it will cause this object be added into those red-black trees or hash structure again.
            FrozenContext.resumeFreezing(ctx);
        }
    }

  1. The user can create an implementation of FrozenComparator<T> for TreeSet<FullName> or TreeMap<FullName, ?> 
    public class FullNameComparator
implements FrozenComparator<FullName> {
    @Override
    public int compare(FullName o1, FullName o2) {
        Integer preCmp = org.babyfish.lang.Nulls.preCompare(o1, o2);
        if (preCmp != null) {
            return preCmp;
        }
        int cmp = org.babyfish.lang.Nulls.compare(o1.getFirstName(), o2.getFirstName());
        if (cmp != 0) {
            return cmp;
        }
        return org.babyfish.lang.Nulls.compare(o1.getLastName(), o2.getLastName());
    }

    @Override
    public void freeze(FullName obj, FrozenContext<FullName> ctx) {
        // "obj" is added into a red-black tree delegated by "ctx",
        // both the "firstNameFrozenContext" and "lastNameFrozenContext" of "obj" 
        // must be changed to let "obj" remember that new red black tree because  
        // the method "compare" of this comparator depends on both 
        // "firstName" and "lastName" of "obj".
        obj.firstNameFrozenContext = FrozenContext.combine(
                obj.firstNameFrozenContext, 
                ctx);
        obj.lastNameFrozenContext = FrozenContext.combine(
                obj.lastNameFrozenContext, 
                ctx);
    }

    @Override
    public void unfreeze(FullName obj, FrozenContext<FullName> ctx) {
        // "obj" is removed from a red-black tree delegated by "ctx",
        // both the "firstNameFrozenContext" and "lastNameFrozenContext" of "obj" 
        // must be changed to let "obj" forget that red black tree because  
        // the method "compare" of this comparator depends on both 
        // "firstName" and "lastName" of "obj".
        obj.firstNameFrozenContext = FrozenContext.remove(
                obj.firstNameFrozenContext, 
                ctx);
        obj.lastFrozenContext = FrozenContext.remove(
                obj.lastNameFrozenContext, 
                ctx);
    }
}

Element crowding out effect

The functionality "Unstable Collection Elements" can cause another instresting issue. As we know, set or map does not allow the duplicated elements or keys, what will be happened when an element or key is changed to new value that is conflict with another element or key? In order to guarantee the uniqueness, BabyFish has to crowd out one of those two duplicated elements or keys, this is called "Element crowding out effect" , it has 2 modes:

  1. Default Mode, retain the current element or key that is modified by user, and crowd out the another conflict element or key. In ObjectModel, only this mode can be used.
  2. Second Mode, retain another conflict element or key, and crowd out the current element or key that is modified by user. This mode can't be used in ObjectModel, it can only be used in X Collection Framework, so it won't be discussed by this document.

Let's continue our example

// so fn3 will be crowded out, that means
// fn3 will be removed from hash structure and red black tree automatically.

Bubbled event of collection ciew Chain

The previous chapter mentioned that BabyFish supports "X Collection Framework" which extends "Java Collection Framework" to support "Unstable Collection Framework" . In order to further support ObjectModel4Java, BabyFish also supports "MA Collection Framwork" which extends "X Collection Framework" . "MA" means "Modification Aware" , MA Collection Framework supports event notification for modifications, it looks like the row level trigger of database very much.

MA Collection Framework can trigger the even notification before modification and after modification, like the database trigger very much, it is very easy to be understood so that it unnecessary to disscuss its tedious details in this document, this document only discuss the most imortant functionality of MA Collection Framework: Bubbled event of collection view chain.

Java Collection Framework is more powerful than the collection framework of other languages, it also means it's more complex. As the most complex functionality, Java Collection Framework supports view, collection object can be used to create view objects, view object can also be used to create deeper view objects. Finally, it's possible to create a view objects tree whose depth is unlimited, the root node of this tree is the original collection object and the other nodes are view objects of that original collection object. All the view objects of this tree share the data with original collection objects, the modification for any object of this tree can affect the other objects. The collection frameworks of other languages either don't support view or support a few views in individual cases, but the view functionality of Java Collection Framework has universality.

Let's look at this example 

// Original collection object, { A: " Alpha" , B: " Beta" , C: " Gamma" , D: " Delta"  }
MANavigableMap<String, String> map = new MATreeMap<>();
map.put("A", "Alpha");
map.put("B", "Beta");
map.put("C", "Gamma");
map.put("D", "Delta");

// Level-1 view object, { D: " Delta" , " C" : " Gamma" , B: " Beta" , A:" Alpha"  }
MANavigableMap<String, String> descendingMap = map.descendingMap();

// Level-2 view object, { " C" : " Gamma" , B: " Beta"  }
MANavigableMap<String, String> subMap = descendingMap.subMap("C", true, "B", true);

// Level-3 view object, [ " B" , " C"  ]
MANavigableSet<String> descendingKeySet = subMap.descendingKeySet();

// Level-4 view object, [" B" ]
MANavigableSet<String> headSet = descendingKeySet.headSet("C", false);

// Level-5 view object, it's a iterator that can iterate " B" .
MAIterator<String> descendingIterator = headSet.descendingIterator();

A chain contains 6 objects was created, the first object is original collection object and other 5 objects are view objects, all the view objects does not have data of themselves but share the data with original collection object, change the data of the deepest view object can affect the original collection object, like this 

descendingIterator.next(); //Let the iterator locate to the element "B" 
descendingIterator.remove(); //Remove the element "B"
After this operation, the original collection object is affected and its new data is { A: "Alpha" , C: "Gamma" , D: "Delta" }. Event notification is triggered by the deepest view object, then it's bubbled to its parent view object to trigger a new event, add so on. Finally, the event will be bubbled to the original collection object to trigger new event, like this

The original collection object itself can always trigger the event notification about the data modification through this functionality, no matter the user modified the collection itself or modified any view object of it, that's why ObjectModel4Java dares to export the collections as public properties.

JPA Part

In practical projects, not only simple memory-only data classes, but also JPA entity classes can be used. JPA is good at do the data exchange between java entity objects and database, but it don't care the the smart bidirectional association management and the consistency of bidirectional association. So ObjectModel4JPA that's extended from ObjectModel4Java is created to support smart JPA/Hibernate entity classes, on the one hand, it let JPA entity classes support smart association, on the other hand, it extends the JPA itself to let JPA become more powerful.

ObjectModel4JPA

ObjectModel4JPA is the advanced extension of ObjectModel4Java for JPA entity classes, in order to In order to maintain compatibility with lazy loading of ORM.

Declare smart entity classes by ObjectModel4JPA

It is worth noting that, be different with ObjectModel4Java, not all the annotations under the package "org.babyfish.model" can be used by ObjectModel4JPA, only these annotations can be used.

  • @org.babyfish.model.jpa.JPAModel
  • Many annotations under the package "javax.persistence" supported by JPA standard
  • @org.babyfish.model.IndexOf
  • @org.babyfish.model.KeyOf
  • @org.babyfish.model.ComparatorRule

  • @org.babyfish.model.Navigable

Important difference between ObjectModel4JPA and ObjectModel4Java
  1. The field of JPA entity class can't be declared as "java.util.SortedSet", "java.util.SortedMap", "java.util.NavigableSet" and "java.util.NavigableMap" becasue of the restriction of JPA/Hibernate; if this is your requirement, please declare the filed as "java.util.Set" or "java.util.Map" and use the annotation "@org.babyfish.model.Navigable".
  2. Generally speaking, it's unnecessary to use the annotation "org.babyfish.model.ComparatorRule" to tell the field whose type is set or map how to calculate the hashCode of set element or map value; Unless the property is "Set" and it used the annotation "@org.babyfish.model.Navigable", or the hashCode calculation rule is your business requirement. That's because the default hashCode calculation rule is very powerful. The default rule is:
    1. First of all, ObjectModelJPA does not allow the type of id(@javax.persistence.Id) property to be primitive type; otherwise, BabyFish will report error.
    2. If the id of current entity object is null, it uses "System.identityHashCode" to calculate the hashCode of element object and check whether two java referances are same to implement the object equality.
    3. If the id of current entity object is not null, it uses the id property value to calculate the hashCode of element object and use equality of id property value to check whether two objects are equal.
    4. If the id of current entity object is changed, all the hash collections and red black tree collections that use the current object will be adjusted automatically because of the functionality "Unstable Collection Elements". So don't worry, please be assured that using the id generation strategy of Hibernate and the configuration "hibernate.use_identifier_rollback".
Bidirectional association of JPA entity classes

JPA entity object is more complex than simpe java object because it supports lazy properties, such as:

  1. Lazy collection for one-to-many or many-to-many property.
  2. Lazy proxy for many-to-one and one-to-one property.
  3. Lazy scalar for basic property with annotation @Basic(fetch = FetchType.LAZY).

The bidirectional association of Hibernate requires one side must real side the other side must be fake side, let's see 2 examples:

  • Exmaple 1, "Employee.department" is real side and "Department.employees" is fake side

    1. The declaration of class "Department": 
      @JPAModel
@Entity
@Table(name = "DEPARTMENT")
public class Department {
    @OneToMany(mappedBy = "department") //Fake side mapped by Employee.department.
    private Set<Employee> employees;
}
    2. The declaration of class "Employee": 
      @JPAModel
@Entity
@Table(name = "EMPLOYEE")
public class Employee {
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "DEPARTMENT_ID") //Real side
    private Department department;
}

  • Example 2, "Department.employees" is real side and "Employee.department" is fake side

    1. The declaration of class "Department": 
      @JPAModel
@Entity
@Table(name = "DEPARTMENT")
public class Department {
    @OneToMany(mappedBy = "department") 
    @JoinColumn(name = "DPERATMENT_ID") //Real side
    private Set<Employee> employees;
}
    2. The declaration of class "Employee": 
      @JPAModel
@Entity
@Table(name = "EMPLOYEE")
public class Employee {
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(
        name = "DEPARTMENT_ID", 
        insertable = false, 
        updatable = false
    ) // Fake side that can be neither inserted nor updated. 
    private Department department;
}

In original Hibernate, only the modification for real side can be used to update the database, if the collection of real side is lazy, it will be loaded immediately when it's changed because the modification for real side is necessary. Of course, exception will be thrown if the session of the lazy collection is lost during the loading process.

In original Hibernate, the modification for fake side is unnecessary, fake side can't be used to change the database even if it has been modified by user. Loading the lazy collection of fake side is huge waste, or it may cause unnecessary exception if the session is lost.

ObjectModel4JPA has the same functionalities with ObjectModel4Java, in order to guarantee the consistency the bidirectional association, BabyFish will modify one side automatically when the other side is modified by the program explicitly, that means both sides are always modified together so that the database will always be updated, no matter which side is changed by programmer. But, is it a problem with huge waste or unnecessary exception?

In order to resolve this problem, ObjectModel4JPA replaces the lazy mechanism of hibernate, for both lazy collections, lazy proxies and lazy scalars. BabyFish supports its own "Hibernate Lazy Collection Framework" which is very cunning. For a unloaded collection of fake side,

  • If it is modified by the program explicitly, it works like the fake side collection with the classic configuration lazy="extra" in "*.hbm.xml", it won't be loaded except using a tiny SQL to check the existence of the modified element in database; but the event of "MA Collection Framework" will still be made up and triggered so that the other real side can be changed by ObjectModel4JPA automactially, finally the database can be updated. It also can remember the change history to let change history can be merged with the data loading in the future. This functionality is called "Lazy modification".
  • If it is modified by ObjectModel4JPA automatically, that means the other real side is changed by the program explicitly so that database will be updated later absolutely. It won't execute any SQL, but the event of "MA Collection Framework" will still be made up and triggered, and it also can remember the change history to let change history can be merged with the data loading in the future. This functionality is called "Dummy modification"

So, don't worry which the real side is and which the fake side is. If any side is changed by programmer, ObjectModel4JPA will change other side automatically and cunningly so that the database will always be updated when the transaction is committed. And don't worry the performance too.

Distrubuted application solution: QueryPath

QueryPath let JPA query support dynamic fetch and dynamic sorting. Not only it can improve the performance by forbiding Hibenrate to generate the SQL with "N + 1" Query, but also it can give a perfect solution for a pain point about lazy initialization error in a distributed application with mutliple JVM instances. This is the most important functionality learned from Active Record and ADO.NET Entity Framework.

QueryPath is combined with 2 parts: FetchPath and SimpleOrderPath, let's talk about QueryPath first.

"org.hibernate.LazyInitializationException" is most popular exception in the actual projects, it means an entity objects that has unloaded lazy properties lost its hibernate session when one lazy property is accessed. In order to resolve this pain point, springframework supports a functionality "Open Session In View".

In the clasic solution "Open Session In View", a Hibernate session will be created when the HTTP request is received and that session will keep alive until the whole HTTP response is generated. During this time, all the web components, business components and data accessing components are running by this session so that any component can load the lazy properties of JPA entity objects very freely. This solution has three disadvantages:

  • It's only friendly for the nested spring transaction whose propagation is REQUIRED.
  • Too many N + 1 query SQL statements will be generated so that the preformance is not good.
  • Largest problem: It can only be used by the simplest application with only one JVM instance, for the distributed application with mulitple JVM instances, this solution is not useful.

Suppose there is a distributed application, business logic components are running on a cluster of application servers, and the web components are running on another cluster of web servers, these two clusters are independent for each other and use the remote invocation to communicate with each other. When the cluster of business logic layer wants to return some data objects to the cluster of web layer, it's necessary to use some remote data transmission technologies such as RMI, SOAP or JSON. Before remote data transmission, the entity objects with lazy properties retain the hibernate session so that lazy properties can be loaded whenever, but after remote transmission, that hibernate session will be lost so that their lazy properties can't be loaded, "org.hibernate.LazyInitializationException" will always be thrown when unloaded lazy properites are accessed by web components.

In order to resolve this problem, BabyFish supports "QueryPath". QueryPath is simple descriptor that is friendly for remote transmission. It's specified by the client side, then it can be sent to server side as a part of query arguments, the server side receives it and it is finally processed by the data accessing object. The data accessing object can parse those descriptors to be fetch plan and merge them into the final generated SQL statements so that all the data object graph required by client-side can be preloaded, the client side needn't to do any work about lazy properties loading. @org.springframework.stereotype.Repository //This is a data accessing object
public class DepartmentRepositoryImpl implements DepartmentRepsitory {

    // This isn't the standard JPA interface "javax.persistence.EntityManager",
    // it's babyfish's interface "org.babyfish.persistence.XentityManager" 
    // which extends "javax.persistence.EntityManager"
    @javax.persistence.PersistenceContext
    private org.babyfish.persistence.XentityManager em;

    @Override
    public List<Department> getDepartmentLikeName(
        String name, //Query condition
        Department__ ... queryPaths //Typed QueryPaths
    ) {
        return this
            .em
            .createQuery("select d from Department d where d.name like :name")
            .setParameter("name", name)
            .setQueryPaths(queryPaths) //Use the "queryPaths" here
            .getResultList();
    }
}

 *    select 
 *        d.DEPARTMENT_ID, d.NAME, d.COMPANY_ID,
 *        e.EMPLOYEE_ID, e.NAME, e.DEPARTMENT_ID,
 *        al.START_TIME, al.END_TIME,
 *        sl.START_TIME, sl.END_TIME,
 *        c.COMPANY_ID, c.NAME
 *    from DEPARTMENT d
 *    left outer join EMPLOYEE e on d.DEPARTMENT_ID = e.DEPARTMENT_ID
 *    left outer join ANNUAL_LEAVE al on e.EMPLOYEE_ID = al.EMPLOYEE_ID
 *    left outer join SICK_LEAVE sl on e.EMPLOYEE_ID = sl.EMPLOYEE_ID
 *    left outer join COMPANY c on d.COMPANY_ID = c.COMPANY_ID
 *    where d.NAME like ?
 *
 *  (2) Fetch the scalar property "description" for all the department objects
 *   (Suppose the first SQL selected 2 department objects)
 *    select d.DEPARTMENT_ID, d.DESCRIPTION
 *    from DEPARTMENT d
 *    where d.DEPARTMENT_ID in (?, ?)
 *
 *  (3) Fetch the scalar property "image" for all the employee objects
 *   (Suppose the first SQL fetched 6 employee objects)
 *    select e.EMPLOYEE_ID, e.IMAGE
 *    from EMPLOYEE e
 *    where e.EMPLOYEE_ID IN (?, ?, ?, ?, ?, ?) 
 */

We've known what the FetchPath is. Now, let's talk about SimpleOrderPath which can be used to sort the query result of the collection association properties, like this 

// Invoke the query method with 5 FetchPaths and 4 SimpleOrderPaths.
List<Department> departments = departmentRepository.getDepartmentsLikeName(
    "%s%",

    /*
     * These 5 QueryPaths start with "begin()" and end with "end()"
     * are called QueryPaths. Be same with previous example
     */
    Department__.begin().description().end(),
    Department__.begin().employees().image.end(),
    Department__.begin().employees().annualLeaves().end(),
    Department__.begin().employees().sickLeaves().end(),
    Department__.begin().company().end()

    /*
     * These 4 QueryPaths start with "preOrderBy()" and end with "asc()" or "desc()" 
     * are called SimpleOrderPaths
     */
    // For the department objects themselves of the query result, 
    // they must be sorted by their name ascending
    Department__.preOrderBy().name().asc(),

    // For each department, the elements of collection property "employees" 
    // must be sorted by their name ascending
    Department__.preOrderBy().employees().names().asc(),

    // For each Employee, the elements of collection property "annualLeaves" 
    // must be sorted by their name ascending
    Department__.preOrderBy().employees().annualLeaves().startTime().desc(),

    // For each Employee, the elements of collection property "sickLeaves" 
    // must be sorted by their name ascending
    Department__.preOrderBy().employees().sickLeaves().startTime().desc()
);

This example uses 9 QueryPaths, the last 4 QueryPaths are SimpeOrderPaths. The first simple order path is very easy to be understood because it's used to sort the root objects themselves of the query result. The next 3 simple order path are used to sort the association properites whose type are collection, but all the collection properties of the JPA classes are declared as "java.util.Set", how could this be? In ObjectModel(Both ObjectModel4Java and ObjectModel4JPA), the collection property whose type is "java.util.Set" or "java.util.Map" are implemented like "java.util.LinkeHashSet" or "java.util.LinkedHashMap" unless its field uses the annotation "@NavigableInstrument". The generic set or map whose order is unpredictable will never be used by ObjectModel4JPA.

Typed QueryPath is very cool, but sometimes it is not easy to specify Typed QueryPath in actual projects. For example, a table on the web page can let user click the column header to specify the dynamic sorting, in this case, the sorted property name and sorting mode can only be sent to web server as string parameter by javascript code of client side. So BabyFish also supports Non-Typed QueryPath whose type is string. Please look at this code(request is a variable of javax.servlet.HttpServletRequest). 

// Get the sorted property name from the javascript ajax request.
String propertyName = request.getParameter("propertyName");

// Get the sorting mode from the javascript ajax request, must be "asc" or "desc"
String orderMode = request.getParameter("orderMode");

// Get the string of query path(Non-typed query path)
String queryPathString = "pre order by this." + propertyName + ' ' + orderMode;

// Compile the string to be typed QueryPath.
Department__[] queryPaths = Department__.compile(queryPathString);
Please view ${babyfish_root_dir}/demo/babyfishdemo-spring/src/test/java/org/babyfishdemo/spring/dal/QueryPathTest.java to know more.
Resolve the defect of paging query for Oracle

Hibernate supports query with "firstResult/maxResults", that's called paging query. Unfortunately, Hibernate's paging query has a defect, but fortunately, if the database is Oracle, it can be resolved by BabyFish. Please don't use the default hibernate dialect for Oracle such as "org.hibernate.dialect.Oracle10gDialect", please use the extended hibernate dialect for Oracle such as "org.babyfish.hibernate.dialect.Oracle10gDialect".

Let's see an simplest example about paging query. 

return this
        .em
        .createQuery("select d from Department d order by d.name asc")
        .setFirstResult(100) // Skip 100 results
        .setMaxResults(10) // Select 10 results at most
        .getResultList();

If the database is Oracle, the generated SQL should be:

select 
    * 
from ( 
    select 
        row_.*, 
        rownum rownum_ 
    from ( 
        select
            d.DEPARTMENT_ID, d.NAME, d.COMPANY_ID 
        from DEPARTMENT d 
        order by d.NAME asc 
    ) row_
    where rownum <= ?
) 
where rownum_ > ?

This is classic Oracle paging query by using the fake column ROWNUM. Hibernate works very good until now. But if we change the code like this

return this
        .em
        .createQuery(
                "select d from Department d" +
                "left join fetch d.employees " + //Collection fetch
                "order by d.name asc"
        )
        .setFirstResult(100) // Skip 100 results
        .setMaxResults(10) // Select 10 results at most
        .getResultList();
and try again, you'll find the generated SQL is 
Select
    d.DEPARTMENT_ID, d.NAME, d.COMPANY_ID,
     e.EMPLOYEE_ID, e.NAME, e.DEPARTMENT_ID
from DEPARTMENT d 
left outer join EMPLOYEE e on d.DEPARTMENT_ID = e.DEPARTMENT_ID
order by d.NAME asc
There is no paging restrictions in this SQL, you can also find a warning message in the Hibernate's log.
firstResult/maxResults specified with collection fetch; applying in memory!

When the JPQL/HQL contains collection fetches, Hibernate has to giveup the SQL-level paging query, it query all the results from database and load all of them into the objects in memory and finally apply the paging filter for those objects. This is a huge performance issue, and the developers are hard to pay attention to that waring message in too.

Why does hibernate have to do this? Because the collection fetch can generate the join to child table so that the data of parent table will be duplicated in the ResultSet, like this.

Data of parent table "DEPARTMENT" Data of child table "EMPLOYEE"
DEPARTMENT_ID NAME EMPLOYEE_ID NAME DEPARTMENT_ID
1 Market 1 Jim 1
1 Market 2 Kate 1
2 Sales 3 Lucy 2
2 Sales 4 Tom 2
In the ResultSet, the left side has duplicated data, but the left side is data required by the user, so applying paging query on the ResultSet is not the expect behavior of user.

Fortunately, If the user changes to use the dialect "org.babyfish.hibernate.dialect.Oracle10gDialect" when the database is Oracle, then try again, the generated SQL become this 

select 
    *
from (
    select
        d.DEPARTMENT_ID, 
        d.NAME, 
        d.COMPANY_ID,
        e.EMPLOYEE_ID,
        e.NAME,
        e.DEPARTMENT_ID,
        dense_rank() over(/* DENSE_RANK is built-in analysis function of Oracle.*/
            order by 
                d.NAME asc, /* Orginal sorted column in order by clause */
                d.ROWID asc /* New sorted column added by BabyFish */
        ) dense_rank____
    from DEPARTMENT d 
    left outer join EMPLOYEE e on d.DEPARTMENT_ID = e.DEPARTMENT_ID
)
where dense_rank____ > ? and dense_rank____ <= ?

BabyFish uses the built-in analysis function "DENSE_RANK" to calculate the rank for each row, several rows with duplicated data of root table will be calculated to be same rank, finally use the rank of each row to do the paging query. Fortunately, it can work normally when the order by clause order only uses the columns of root table.

Next, let's see the the most unfortunate case, change the java code like this:

return this
        .em
        .createQuery(
                "select d from Department" +
                "left join fetch d.employees " + /* Collection Fetch */
                "order by " +
                "  d.name asc, " + /* Not only order by the column of root table */.
                "  e.name asc" /*, but also order by the column of child table.*/
        )
        .setFirstResult(100) // Skip 100 results
        .setMaxResults(10) // Select 10 results at most
        .getResultList();

Now, the order by clause of JPQL/HQL uses not only the column(s) of root table, but also the column(s) of other table(s). BabyFish has to uses another analysis function DISTINCT_RANK(ROWID), like this: 

select 
    *
from (
    select
        d.DEPARTMENT_ID, 
        d.NAME, 
        d.COMPANY_ID,
        e.EMPLOYEE_ID,
        e.NAME,
        e.DEPARTMENT_ID,
        distinct_rank(d.ROWID) over(
            order by 
                d.NAME asc, 
                e.NAME asc
        ) distinct_rank____
    from DEPARTMENT d 
    left outer join EMPLOYEE e on d.DEPARTMENT_ID = e.DEPARTMENT_ID
)
where distinct_rank____ > ? and distinct_rank____ <= ?

"DISTINCT_RANK" is an user defined analysis function, it's defined by BabyFish. In order to make sure that user defined analysis function will be created in database, you have 2 choices

  • Automatic installation: Configure the JPA/Hibernate property "babyfish.hibernate.create_oracle_distinct_rank" to be "true" in persistence.xml or spring configuration files.
  • Mannual installation: Execute "oracle_distinct_rank_java.sql" and "oracle_distinct_rank.sql" under the directory "${babyfish_root_dir}/install/oracle/"
Please view ${babyfish_root_dir}/demo/babyfishdemo-spring/src/test/java/org/babyfishdemo/spring/dal/PagingQueryForDistinctModeTest.java to know more.
